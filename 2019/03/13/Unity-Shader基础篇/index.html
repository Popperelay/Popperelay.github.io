图形程序员，是程序员中的“外貌协会”！

OpenGL把屏幕的左下角当成最小的窗口坐标值，而DirectX是左上角（因为微软的窗口都使用了这样的坐标系统，毕竟和我们的阅读方式相同：从上到下，从左到右）。

片元着色器一般不可以访问到邻近片元的信息，有个例外：片元着色器可以访问到导数信息。

如果在片元着色器之前进行深度测试（Early-Z），虽然会减少很多不不必要的片元计算，但是有可能和片元着色器中的一些操作冲突。比如我们在片元着色器里手动做了透明度测试，如果做了前置深度测试，就有可能会导致透明的结果出现问题。

现代的GPU会判断片元着色器中的操作是否和提前深度测试发生冲突，如果有冲突，就会禁用前置深度测试。但是这样也会造成性能上的下降，因为有更多片元需要被处理了。这也是透明度测试会导致性能下降的原因。

CPU会向命令缓冲区中添加命令（改变渲染状态的命令以及DrawCall），而GPU会从命令缓冲区中取出命令进行执行，从而使得CPU和GPU可以并行工作。

但是DrawCall多了为何会影响效率呢？
因为在每次调用DrawCall时，CPU都需要做很多准备工作（比如改变渲染状态），这些准备工作是很费时的；同时GPU的渲染能力又很强，渲染200个还是2000个三角面片通常没有什么区别，所以GPU的渲染速度往往快于CPU提交命令的速度。如果DrawCall数量太多，CPU就会把大量时间花费在DrawCall的准备工作上，造成CPU的过载和GPU等待浪费。

那么怎么减少DrawCall呢？
有一个方法是批处理：把很多小的DrawCall合并成一个大的DrawCall。实际做法就是在CPU的内存中合并网格。不过合并的过程是需要消耗时间的，所以批处理技术更加适合于那些静态的物体，例如不会移动的大地、石头等等，对于这些静态物体我们只需要合并一次即可。当然也可以对动态物体进行批处理合并，但是由于它们是不断运动的，所以每一帧都需要重新合并再发送给GPU，这种每帧都重新合并的消耗恐怕还不如直接调用多次DrawCall~。

在Unity的SubShader中进行的设置，会用于所有的Pass。

SubShader的标签(Tags)是一个键值对，它的键和值都是字符串类型，用来告诉Unity的渲染引擎：我希望怎样以及何时渲染这个对象。如下：

Tags {"TagName1" = "Value1" "TagName2" = "Value2"}

Unity内部会把所有Pass的名称转换成大写字母的表示，因此在使用UsePass命令进行复用代码时，必须使用大写形式的名字：

UsePass "MyShader/MYPASSNAME"

Unity还支持一些特殊的Pass，比如GrabPass：该Pass负责抓取屏幕并将结果存储在一张纹理中，以用于后续的Pass处理。

Fallback还会影响阴影的投射。因为Unity在渲染阴影纹理时，会在每个Unity Shader中寻找一个阴影投射的Pass，通常情况下，Fallback使用的内置Shader中包含了这样一个通用的Pass：

Fallback "VertexLit"

表面着色器（Surface Shader）被定义在SubShader语义块（而非Pass语义块）中的CGPROGRAM和ENDPROGRAM之间，因为表面着色器不需要开发者关心使用多少个Pass、每个Pass如何渲染等问题，Unity会在背后为我们做好这些事情，我们只需要传给它需要的纹理等等。

对于模型空间和世界空间，Unity使用的是左手坐标系，但是对于观察空间，Unity使用的是右手坐标系。

向量叉积的模等于两个向量的模和夹角正弦的乘积：$|a\times b|=|a||b|sin\theta$。

仿射变换（affine transform）就是合并线性变换和平移变换的变换类型。仿射变换可以使用一个4x4的矩阵来表示，为此，需要把矢量扩展到四维空间下，这就是齐次坐标空间（homogeneous space）。

对于法线向量，为了使其模型变换以后，法线依然垂直表面，那么法线应该左乘的不是模型矩阵，而是模型矩阵的逆转置矩阵。

如果只包括旋转和统一缩放（假设缩放系数是k），那么变换矩阵（在Unity里是UNITY_MATRIX_MV）就几乎是一个正交矩阵了。为什么是几乎呢？因为统一缩放可能会导致每一行（或每一列）的矢量长度不为1，而是k，这不符合正交矩阵的特性，但我们可以通过除以这个统一缩放系数，来把它变成正交矩阵。

对于向量$v$来说，左乘正交矩阵$M$的逆（或者转置），等效于向量$v$的转置来右乘矩阵$M$，再将得到的向量进行转置。因为：$(v^{T}M)^{T}=M^Tv=M^{-1}v$。所以变换法线向量时通常使用如下代码（最后一步结果向量的转置是Unity自动做的，不需要额外写代码显式转置）：

o.normal = mul(v.normal, (float3x3)unity_WorldToObject);

对于Unity Shader中mul函数（源自于cg语言），mul会根据向量是左乘还是右乘矩阵，来自动将向量变换为行或者列矩阵（或者说行/列向量）：

float col_mul_result = mul(M, v);  //向量v被转换为列矩阵
float row_mul_result = mul(v, M);  //向量v被转换为行矩阵

如果要自己写数组去填充Unity Shader中的矩阵，需要注意的是它使用的按行存储的方式（因为Cg语言是这么规定的）。但是，在Unity的脚本里提供了另一种矩阵类型：Matrix4x4，它是按列存储的，与Shader中的不一样。
