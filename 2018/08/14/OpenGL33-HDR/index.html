为何需要HDR

一般来说，当存储在帧缓存中时，亮度和颜色的值会被默认限制在0.0到1.0之间。很多时候，这个规则不会影响到最终的视觉效果。但是，如果我们遇上了一个特定的区域，其中有多个光源叠加在一起，使得颜色值总和超过了1.0，会发生什么？答案是颜色值或者亮度超过1.0的会被截断为1.0（这是显示器限制的），从而导致场景会有一片什么也看不清的白色，丢失场景细节。如下图所示：

这是由于大量片段的颜色值都非常接近1.0，在很大一个区域内大多数片元都被截断为相同的白色（即1.0），导致损失了很多细节，使场景看起很假。

一个解决方案是减小光源强度，从而保证场景内没有一个片元的颜色（或者亮度）会超过1.0。显然，这不是一个好的解决方案，因为需要使用不切实际的光照参数。

另一个更好的解决方案是允许颜色值暂时超过1.0，最后再将其转换至0.0到1.0的区间内，从而防止损失细节。

允许颜色值超过1.0，有更大的颜色范围，这就是HDR（High Dynamic Range，高动态范围）。

HDR原理和色调映射

HDR原本是被运用在摄影上，摄影师对同一个场景采取不同曝光拍多张照片，捕捉大范围的色彩值。这些图片被合成为HDR图片，从而综合不同的曝光等级使得可以看到大范围的细节。

HDR渲染类似，我们允许用更大范围的颜色值从而获取大范围的黑暗与明亮的场景细节，最后将所有HDR值转换成在[0.0,1.0]之间的LDR（Low Dynamic Range，低动态范围）。

转换HDR值到LDR值的过程叫做色调映射（Tone Mapping）。如今又很多的色调映射算法，这些算法致力于在转换过程中能够保留尽可能多的HDR细节。简单地平均值重新转换这些颜色值并不能很好的解决这个问题，因为明亮的地方会显得更加显著。我们能做的是用一个不同的函数曲线（或函数方程）来完成从HDR值到LDR值的转换，从而完全掌控场景的亮度。

浮点帧缓冲

当帧缓冲的颜色附件使用类似GL_RGB这种标准化的定点格式时，OpenGL会把存入其中的颜色值自动约束到0.0到1.0之间。

而HDR要求能够存储在0到1范围之外的浮点颜色值。我们可以把颜色附件的格式设置为类似GL_RGB16F、GL_RGBA16F等等，这时候的帧缓存叫做浮点帧缓冲，浮点帧缓冲可以存储超过0.0到1.0范围的浮点值，所以非常适合HDR渲染（除非需要非常高的精度通常16位就够了，不需要使用32的GL_RGB32F）。

这样我们就可以先把场景颜色值渲染到浮点帧缓冲的颜色附件上，然后再渲染一张铺屏四边形（Screen-filling Quad），把该颜色附件的通过色调映射转换到0到1之间，再显示到屏幕上。代码类似下面这样子：

glBindFramebuffer(GL_FRAMEBUFFER, hdrFBO);
    glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);  
    // [...] 渲染(光照的)场景
glBindFramebuffer(GL_FRAMEBUFFER, 0);

// 现在使用一个不同的着色器将HDR颜色缓冲渲染至2D铺屏四边形上
hdrShader.Use();
glActiveTexture(GL_TEXTURE0);
glBindTexture(GL_TEXTURE_2D, hdrColorBufferTexture);
RenderQuad();

其中我们渲染的场景是一个被4个点光源照亮的拉伸的立方体通道，其中一个非常亮的点光源在通道尽头：

std::vector<glm::vec3> lightColors;
lightColors.push_back(glm::vec3(200.0f, 200.0f, 200.0f));
lightColors.push_back(glm::vec3(0.1f, 0.0f, 0.0f));
lightColors.push_back(glm::vec3(0.0f, 0.0f, 0.2f));
lightColors.push_back(glm::vec3(0.0f, 0.1f, 0.0f));

把场景颜色值渲染到浮点帧缓冲的着色器程序很简单，和以前渲染到普通帧缓冲是一样的，就不再赘述了。渲染铺屏四边形，把浮点颜色附件里的颜色值显示到屏幕上的片元着色器程序如下：

#version 330 core
out vec4 color;
in vec2 TexCoords;

uniform sampler2D hdrBuffer;

void main()
{             
    vec3 hdrColor = texture(hdrBuffer, TexCoords).rgb;
    color = vec4(hdrColor, 1.0);
}

运行结果如下图：

可以看到还是和以前一样，亮的地方依然看不清细节。这是因为上面的片元着色器中并没有做色调映射，只是简单的将纹素值输出给默认帧缓冲，这时默认帧缓冲还是会把超过1.0的颜色值截断为1.0。所以我们需要一些色调映射算法。

色调映射

色调映射是一个损失很小的转换浮点颜色值到我们所有的LDR[0.0,1.0]范围内的过程。

Reinhard色调映射

最简单的色调映射算法是Reinhard色调映射，每一个HDR颜色值都会有一个对应的LDR颜色值，该算法平均地将所有亮度值分散到LDR上。不多说了，算法很简单，直接看代码吧：

void main()
{             
    const float gamma = 2.2;
    vec3 hdrColor = texture(hdrBuffer, TexCoords).rgb;

    // Reinhard色调映射
    vec3 mapped = hdrColor / (hdrColor + vec3(1.0));
    // Gamma校正
    mapped = pow(mapped, vec3(1.0 / gamma));

    color = vec4(mapped, 1.0);
}

运行结果如下：

可以看到应用了Reinhard色调映射之后，通道尽头的木头纹理变得可见了，我们不再会在场景明亮的地方损失细节。当然，这个算法是倾向于明亮的区域的，暗的区域会不那么精细也不那么有区分度。

曝光色调映射

另一个有趣的色调映射算法使用了曝光（Exposure）参数。如果我们有一个场景要展现日夜交替，那么我们当然会在白天使用低曝光，在夜间使用高曝光，就像人眼调节方式一样。

一个简单的曝光色调映射算法会像这样：

uniform float exposure;

void main()
{             
    const float gamma = 2.2;
    vec3 hdrColor = texture(hdrBuffer, TexCoords).rgb;

    // 曝光色调映射
    vec3 mapped = vec3(1.0) - exp(-hdrColor * exposure);
    // Gamma校正 
    mapped = pow(mapped, vec3(1.0 / gamma));

    color = vec4(mapped, 1.0);
}

高曝光值会使通道的黑暗部分显示更多的细节，而低曝光值会显著减少黑暗区域的细节，但会增加更多明亮区域的细节。下面这组图展示了在不同曝光值下的场景：

这个图片清晰地展示了HDR渲染的优点：在正常曝光下木头纹理结构隐约可见，但使用低曝光木头纹理就可以清晰可见了，而使用高曝光，可以更清楚地看到近处的木头纹理。

HDR扩展

前面展示的两个算法仅仅是大量（更高级）的色调映射算法的一小部分，这些算法各有长短。一些色调映射算法倾向于特点的某种颜色/强度，有一些算法能够同时显示低曝光和高曝光下的颜色，从而显示更加多彩和精细的图像。还有一些自动曝光调整的色调映射算法，它能够检测前一阵场景的亮度并且缓慢调整曝光参数，使得场景在黑暗区域逐渐变亮，在明亮区域逐渐变暗。

总结一下HDR渲染的一些优点：在明亮和黑暗区域无细节损失，因为它们可以通过色调映射来恢复；多个光源叠加下不会导致亮度被截断的区域；光源可以被设定为它们原来的亮度而不是被LDR值限定；GDR也使得一些有趣的效果更加可行和真实（比如泛光（Bloom））。
